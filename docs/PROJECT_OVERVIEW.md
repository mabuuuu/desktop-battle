# Desktop Battle - 项目总览

## 项目名称

**Desktop Battle**（桌面火柴人大乱斗）

## 项目愿景

在 Windows 10/11 桌面上实现多阵营火柴人自动战斗模拟。火柴人以线条形式在任务栏（地面）上活动，可攀爬窗口，采集资源、建造建筑、制造武器，最终与敌对阵营交战。整个战斗过程全自动进行，用户以观赏为主，后期支持鼠标交互。

## 核心玩法

### 游戏流程

```
开局 → 采集资源 → 建造工具台 → 制作武器 → 建造兵营 → 生产单位 → 探索 → 发现敌人 → 战斗
```

### 核心机制

1. **资源采集**：每个阵营出生时拥有一个资源采集点，单位前往采集点获取资源
2. **制作链**：资源 → 工具台 → 武器/工具，不同武器需要不同资源和工具台等级
3. **建筑系统**：兵营持续生产单位，其他建筑提供不同功能
4. **探索与战斗**：单位在桌面上探索，发现敌方后进行战斗
5. **观赏性优先**：高血量 + 低伤害，保证长时间战斗的观赏体验

### 地形系统

| 元素 | 对应桌面 | 说明 |
|------|---------|------|
| 地面 | Windows 任务栏 | 火柴人的主要活动区域 |
| 平台 | 桌面窗口 | 可攀爬、站立的平台 |
| 天空 | 桌面空白区域 | 无支撑时受重力下落 |
| 采集点 | 特定位置标记 | 阵营出生点附近 |

## 阵营

初期两个阵营，需在视觉上有明确区分：

| 阵营 | 主色 | 辅色 | 风格 |
|------|------|------|------|
| 红方 (Red Faction) | #FF4444 | #FF8866 | 激进、攻击偏向 |
| 蓝方 (Blue Faction) | #4488FF | #66AAFF | 稳健、防御偏向 |

阵营差异仅为外观和微调属性倾向，不产生根本性玩法差异，保证公平性。

## 文档驱动开发流程

### 核心原则

**所有改动先改文档，后改代码。** 代码必须与文档保持一致。

### 流程

```
需求变更/新功能
    ↓
修改设计文档（明确规格）
    ↓
更新 PLAN.md（记录本次改动目标）
    ↓
更新 CHANGELOG.md（记录变更内容）
    ↓
实现代码
    ↓
验证代码与文档一致
    ↓
Git 提交并 Push（提交信息引用文档变更）
```

### 文档体系

| 文档 | 职责 | 更新时机 |
|------|------|---------|
| `PROJECT_OVERVIEW.md` | 项目愿景、核心玩法、流程 | 需求变更时 |
| `TECHNICAL_SPEC.md` | 技术栈、架构、渲染、物理 | 技术方案变更时 |
| `UNIT_DESIGN.md` | 单位属性、外观、阵营区分 | 单位相关变更时 |
| `WEAPON_DESIGN.md` | 武器类型、制作链、伤害 | 武器相关变更时 |
| `BUILDING_DESIGN.md` | 建筑类型、功能、资源消耗 | 建筑相关变更时 |
| `BEHAVIOR_DESIGN.md` | 行为树、AI决策、行为定义 | AI/行为变更时 |
| `PHYSICS_DESIGN.md` | 物理、碰撞、攀爬、地面 | 物理系统变更时 |
| `LOGGING_DESIGN.md` | 日志规范、格式、存储 | 日志系统变更时 |
| `CHANGELOG.md` | 变更记录 | 每次变更时 |
| `PLAN.md` | 阶段计划、每次改动目标 | 计划调整时 |

## 项目范围

### 初期（V1）

- 2个阵营，各有出生点和资源采集点
- 资源采集 → 工具台 → 武器制作链
- 兵营生产单位
- 任务栏地面 + 窗口平台攀爬
- 自动战斗，纯观赏
- 行为日志、系统日志、错误日志

### 后期（V2+）

- 鼠标交互（选中、指挥）
- 更多阵营
- 更多建筑和武器类型
- AI策略层（LLM指挥官）
- 训练强化学习模型
